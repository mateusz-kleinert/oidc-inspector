# OpenID Connect Flows

## What is OpenID Connect?

OpenID Connect (OIDC) is an identity layer built on top of OAuth 2.0 (RFC 6749). It defines how clients can verify the identity of an end-user and obtain basic profile information in an interoperable way.

- **Core specification**: <https://openid.net/specs/openid-connect-core-1_0.html>
- **Discovery**: <https://openid.net/specs/openid-connect-discovery-1_0.html>
- **OAuth 2.0 base**: <https://www.rfc-editor.org/rfc/rfc6749>

## Supported Flows

### 1. Authorization Code Flow + PKCE (`--flow pkce`)

**Recommended for public clients** (SPAs, native apps, CLIs).

PKCE (Proof Key for Code Exchange, RFC 7636) prevents authorization code interception attacks by binding the code to a secret only the originating client knows.

```
Client                        Browser / OS              Authorization Server
  │                               │                             │
  │── generate code_verifier ─────│                             │
  │── code_challenge = S256(v) ───│                             │
  │                               │                             │
  │── Open browser ──────────────►│                             │
  │                               │── GET /authorize ──────────►│
  │                               │   ?client_id=…              │
  │                               │   &response_type=code       │
  │                               │   &code_challenge=…         │
  │                               │   &code_challenge_method=S256│
  │                               │   &state=…  &nonce=…        │
  │                               │                             │
  │                               │◄── Login page ─────────────│
  │                               │── User authenticates ──────►│
  │                               │                             │
  │                               │◄─ 302 → redirect_uri?       │
  │◄─ Callback received ──────────│   code=…&state=…            │
  │                               │                             │
  │── verify state == original ───│                             │
  │                               │                             │
  │───────── POST /token ────────────────────────────────────────►│
  │          grant_type=authorization_code                       │
  │          code=…  code_verifier=…  client_id=…               │
  │                                                              │
  │◄────────── access_token + id_token + refresh_token ──────────│
  │                                                              │
  │───────── GET /userinfo (Bearer access_token) ───────────────►│
  │◄────────── user claims ─────────────────────────────────────│
```

**Parameters generated by the client:**

| Parameter | Description |
|-----------|-------------|
| `code_verifier` | Random 43–128 char URL-safe string (RFC 7636 §4.1) |
| `code_challenge` | `BASE64URL(SHA256(ASCII(code_verifier)))` |
| `code_challenge_method` | Always `S256` |
| `state` | Random opaque value for CSRF protection (RFC 6749 §10.12) |
| `nonce` | Random opaque value; embedded in `id_token` to prevent replay attacks |

### 2. Authorization Code Flow (`--flow code`)

**For confidential clients** (server-side web apps) that can securely store a client secret. Identical to the PKCE flow but without `code_challenge`; the client authenticates at the token endpoint using its secret.

```
Client                        Browser / OS              Authorization Server
  │                               │                             │
  │── Open browser ──────────────►│                             │
  │                               │── GET /authorize ──────────►│
  │                               │   ?client_id=… &state=…     │
  │                               │── User authenticates ──────►│
  │◄─ Callback: code=…&state=… ──│                             │
  │                               │                             │
  │───── POST /token ───────────────────────────────────────────►│
  │      grant_type=authorization_code                           │
  │      code=…  client_id=…  client_secret=…                   │
  │◄──── access_token + id_token ───────────────────────────────│
```

---

### 3. Client Credentials Flow (`--flow client_credentials`)

**Machine-to-machine** — no user interaction. The client authenticates directly with its credentials to obtain an access token on behalf of itself.

```
Client                                      Authorization Server
  │                                               │
  │─── POST /token ──────────────────────────────►│
  │    grant_type=client_credentials              │
  │    client_id=…  client_secret=…  scope=…     │
  │◄── access_token ──────────────────────────────│
```

No `id_token` is issued (no user context). The `/userinfo` endpoint is typically not available for this flow.

## Token Types

| Token | Format | Purpose |
|-------|--------|---------|
| `access_token` | JWT or opaque | Sent to resource servers as a Bearer token |
| `id_token` | Always JWT | Carries user identity claims; consumed by the client |
| `refresh_token` | Opaque | Exchanges for new access tokens without re-authentication |

---

## Security Considerations

- Always use PKCE for public clients — it is required by [RFC 9700](https://www.rfc-editor.org/rfc/rfc9700) (OAuth 2.1).
- Validate the `state` parameter on every callback to prevent CSRF.
- Validate the `nonce` claim in the `id_token` to prevent replay attacks.
- Validate `iss`, `aud`, `exp`, `iat` claims in the `id_token` before trusting it.
- Never use the `implicit` flow — it is deprecated in OAuth 2.1.
- Prefer short-lived access tokens and use refresh tokens.
- Store client secrets only server-side; never ship them in browser or native-app binaries.
